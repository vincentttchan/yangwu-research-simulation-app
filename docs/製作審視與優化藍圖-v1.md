# 《自強三十年》探索版製作審視與優化藍圖 v1

> 日期：2026-06-09  
> 範圍：遊戲機制、玩法節奏、視覺藝術、UI/UX、互動穩定性、前端工程。  
> 不含：研究 logger、後台、登入、Supabase、部署、倫理文件。

## 1. 製作定位

《自強三十年》探索版現時已不再是單純原型。它已具備：

- 登陸頁、人物路線、歷史過場、地圖探索、城市場景、事件選擇、角色追問、證據任務、懸案、四方阻力、見識、人物冊、功業錄、結局轉場。
- 完整的洋務運動時序和 DSE 學習內容。
- 明確的視覺方向：晚清、劫火、古籍、地圖、朱砂、宣紙、檔案感。
- 可支援研究的潛在互動資料結構。

下一階段的核心不是「再加功能」，而是把現有內容收斂成一個穩定、沉浸、可展示、可研究的 vertical slice。

**製作目標：**
玩家在 5 分鐘內清楚感到：

1. 我是晚清求強局勢中的一名見證者／參與者。
2. 我要在地圖中追查城市、史事、人物和證據。
3. 我的抉擇會影響時局阻力、見識與日後結局。
4. 我不是在做 worksheet，而是在歷史局勢中行走。

## 2. 目前最大優勢

### 2.1 歷史內容密度高

洋務運動常見教材容易變成「成敗原因列表」。本遊戲已把技術、制度、財政、外交、民情、保守勢力、人物選擇等放入互動流程，具備學習歷史複雜性的基礎。

### 2.2 氣氛與主題辨識度強

序章的圓明園劫火、舊金字體、朱砂牌、晚清地圖和宣紙質感已形成強烈第一印象。這是本作最有記憶點的美術資產。

### 2.3 系統已能支撐再玩

四條路線、人物冊、功業錄、懸案、條件選項、不同結局評語，已經有 replay value 的雛形。

### 2.4 教育研究價值高

玩家的移動、證據任務、選項、追問、人物結識、懸案處理和結局結果，都可轉化為 evidence use、historical empathy、argumentation 和 historical complexity 的資料來源。

## 3. 核心問題診斷

### 3.1 第一分鐘仍有互動阻力

問題：
- 登陸頁、人物選擇、過場、地圖、教學提示之間仍偶爾出現重複交代。
- 右下角功能曾被隱藏 screen 攔截，反映 screen layer 規則仍需制度化。
- 玩家未必立即分辨「主線目標」「可探索城市」「未解鎖城市」「功能入口」。

風險：
第一分鐘是學生是否願意繼續玩的決定點。若此處像測試頁或 UI 堆疊，沉浸感會被打斷。

優化方向：
- 保留電影感，但減少中間說明卡。
- 以目標城市、地圖脈衝、史事卷宗和簡短 coachmark 承擔引導。
- 建立 screen/modal interaction rules，防止隱藏層攔截。

### 3.2 核心玩法動詞過多

目前玩家可做的事很多：

- 選人物
- 去城市
- 點熱點
- 讀觀察
- 做證據任務
- 探訪事件
- 回答角色追問
- 處理懸案
- 結識人物
- 看人物冊／功業錄
- 推至要事
- 看結局

問題不是功能錯，而是缺一個清楚的「主循環語法」。

建議把核心循環明確定為：

**看局勢 → 赴城市 → 察線索 → 入事件 → 作抉擇 → 承後果**

其他系統都應服務這條主循環：

- 人物冊：記錄旅途中遇見的人。
- 功業錄：記錄旅途成果。
- 懸案：製造時間壓力與歷史阻力。
- 見識：表示玩家看懂了多少層歷史。
- 四方阻力：表示改革被哪些結構力量牽制。

### 3.3 前台鷹架仍需一致地隱性化

已完成很多顯性 scaffold 移除，但仍要持續守住原則：

- 玩家不應看見「DSE：」「器物／制度／思想分類」「步驟標籤」等老師語言。
- 玩家應透過畫面、角色語氣、選項後果、事件連鎖去理解。
- 研究與教師分析需要的分類應留在資料層，而非前台。

判斷準則：
若一段 UI 文字像試卷題目、筆記標題或教師講義，它應該被改寫成角色話語、局勢描述或玩家行動。

### 3.4 視覺語言有多個版本疊加

目前視覺大致分成幾組：

- 序章：電影海報、劫火、舊金字。
- 人物選擇：卡牌、肖像、能力面板。
- 地圖：策略地圖、印章、四方阻力面板。
- 城市場景：半透明觀察、熱點、bottom sheet。
- 事件／證據／手卷：宣紙、古籍、朱砂。

各自有亮點，但部分界面仍像不同 sprint 的產物。

統一方向：
**晚清檔案美學**。

關鍵語彙：
- 地圖：行軍／輿圖。
- 事件：卷宗。
- 證據：批註／摘錄。
- 人物：名冊。
- 成就：功業錄。
- 結局：回顧／史評。

要避免：
- 現代 dashboard 感過強。
- 裝飾性紙張太多導致畫面笨重。
- 每個 modal 都像不同設計系統。

### 3.5 手機/iPad 體驗仍需獨立設計

桌面地圖和右側資料欄不等於 iPad／手機可玩。

風險：
- 手機上地圖、事件列表、四方阻力、城市 sheet 容易擠壓。
- 文字密度高，學生在手機上會快速疲勞。
- 點擊熱點需要 44px 以上 hit area，但視覺點要小。

方向：
- iPad：保留地圖主視覺，右側資料欄可收窄或摺疊。
- 手機：Map-first + Bottom Mission Sheet，資料欄改為抽屜或短摘要。
- 所有 modal 應有手機專屬排版，不只是縮小桌面。

### 3.6 工程架構已接近拆分臨界點

現況：

- `src/intro.js` 約 7,600 行。
- `src/style-explore.css` 約 8,000 行。
- 遊戲資料、狀態、UI、事件、成就、人物、debug、音效、存檔都集中於大型檔案。

風險：
- 小修容易影響遠處功能。
- CSS 優先級逐層疊加，隱藏 screen 攔截點擊就是典型症狀。
- 後續接研究 logger 或 Supabase 時，若不先穩定事件 ID 與狀態流，會變得難維護。

短期方向：
- 不大重構，先建立穩定性規則和測試。
- 把「screen/modal layer policy」寫入 CSS 和 stability checks。
- 新增 UI 前必須先確認 z-index、pointer-events、hidden 行為。

中期方向：
- 拆出 `state.js`、`routes.js`、`map.js`、`city.js`、`events.js`、`gallery.js`、`audio.js`。
- CSS 按 screen/module 拆分或至少建立區塊索引。

## 4. 優化工程總表

### Sprint A：Interaction Stability

目標：先讓遊戲不再被隱藏層、舊 overlay、拖曳 ghost、未關閉 modal 破壞。

工作：
- 建立全域 screen hidden 規則：hidden screen 和其子元素不得接收互動。
- 檢查所有 fixed overlay 是否有 `[hidden]{display:none}`。
- 統一 modal 開關：open/close/backdrop/ESC。
- 補 stability checks：右下角入口、screen hit target、overlay hidden。
- 禁止圖片原生拖曳，避免人物圖 ghost。

驗收：
- 登陸頁三入口可點。
- 人物選擇頁不再覆蓋登陸頁。
- 任一 hidden overlay 不攔截點擊。
- 桌面／iPad／手機至少各做一次 smoke test。

### Sprint B：First 5 Minutes Polish

目標：把「第一次玩」打磨成可展示 vertical slice。

範圍：
登陸頁 → 選李鴻章 → 四拍過場 → 地圖 → 第一個目標城市 → 第一個證據任務 → 第一個事件。

工作：
- 刪除所有重複任務說明。
- 讓玩家第一步由地圖目標引導，而非長教學。
- 檢查 coachmark 語氣，改為幕僚提示／局勢低語。
- 確保第一個城市有清楚的紅點、短觀察、證據任務、事件結果。
- 減少第一輪非必要入口的視覺干擾。

驗收：
- 5 分鐘內玩家能完成第一個事件。
- 不需要老師口頭補充才知道下一步。
- 前台不出現 DSE 標籤、顯性分類或 worksheet 語氣。

### Sprint C：Core Loop Clarity

目標：把玩法語法固定成「赴城—察證—入事—抉擇—承後果」。

工作：
- 地圖左上要務只保留最重要資訊。
- 城市 bottom sheet 只顯示當前可做的 1–2 個行動。
- 證據任務完成後直接銜接事件或回城市，不再多頁面。
- 事件結果顯示後果，不顯示教學分類。
- 四方阻力變化以敘事和視覺語氣呈現，數值只在必要處保留。

驗收：
- 玩家每次只需回答：「我下一步去哪裏／點哪個線索／作哪個選擇？」
- 視覺上沒有同時出現三個以上行動焦點。

### Sprint D：Visual System Unification

目標：把所有頁面收斂到「晚清檔案美學」。

工作：
- 定義核心視覺 token：宣紙、朱砂、舊金、墨黑、地圖灰綠。
- 統一 modal：卷宗類、名冊類、結局類三種，不再每個 modal 自成一格。
- 地圖 UI 減少紙卡厚重感，保留策略清晰度。
- 城市場景提示改為半透明批註，而非大紙卡。
- 功業錄／人物冊改為「藏書閣／名冊」感，避免像現代成就面板。

驗收：
- 任意截圖都能看出同一個遊戲的語言。
- UI 不再像多輪 sprint 疊加。

### Sprint E：Mobile and iPad Production Pass

目標：不是讓桌面縮小，而是做真正的觸控版。

工作：
- iPad 橫向：Map-first，側欄可摺疊。
- 手機直向：地圖主畫面 + bottom sheet + 抽屜式史事／阻力。
- 所有按鈕 hit area ≥ 44px。
- 城市熱點視覺點小、hit area 大。
- modal 高度不超過視窗，內容可安全捲動。

驗收：
- 1024×768、820×1180、390×844 均可完成第一個事件。
- 文字不溢出，不遮擋主互動。

### Sprint F：Content and Learning Pass

目標：讓歷史學習更像「看懂局勢」，不是「答對知識點」。

工作：
- 每個主要事件檢查：是否有歷史複雜性，而非單因果。
- 每個角色追問檢查：是否像人物／幕僚追問，而非測驗題。
- Transfer Task 對應元素留在後台和研究資料，不前台硬標。
- 檢查洋務失敗原因是否能自然累積：技術、制度、財政、保守派、民智、官督商辦、外交壓力。

驗收：
- 玩完後學生能說出「洋務不是只因一個原因失敗」。
- 不需要在前台顯示「制度／思想／器物」也能形成理解。

### Sprint G：Technical Modularisation

目標：降低後續維護風險。

工作：
- 先不大拆，只新增註解索引和 module map。
- 抽出純資料表：cities、events、persons、achievements、suspenses。
- 把 gallery 開關、screen 狀態、modal 狀態變成共用 helper。
- 保留 stability checks 作為每次 UX 修正的護欄。

驗收：
- 新增一個事件或城市不需要在 7,000 行 JS 中搜尋多處。
- UI 層級 bug 可由測試先捕捉。

## 5. 建議優先序

### 立即做

1. Sprint A：Interaction Stability。
2. Sprint B：First 5 Minutes Polish。

理由：
這兩項直接影響試玩、展示、學生第一印象，也會阻止更多 bug 滾大。

### 接着做

3. Sprint C：Core Loop Clarity。
4. Sprint D：Visual System Unification。

理由：
現有內容已足夠，需要把「多」整理成「清楚」。

### 技術部署前後同步做

5. Sprint E：Mobile/iPad。
6. Sprint G：Technical Modularisation Lite。

理由：
部署與研究收數前，互動和狀態必須穩。模組化不必一次做完，但要先停止大型單檔繼續膨脹。

## 6. 下一個建議 sprint

**Sprint A1：Screen and Modal Interaction Rules**

範圍：
- screen hidden/pointer-events 規則。
- overlay `[hidden]` 規則盤點。
- 登陸頁功能入口 hit target 測試。
- 圖片拖曳 ghost 防護。
- gallery/modal 開關 smoke test。

這個 sprint 已因 2026-06-09 右下角 bug 開始處理，下一步應擴展成全站互動穩定性護欄。

完成 A1 後，才進入：

**Sprint B1：First 5 Minutes Playthrough Polish**

聚焦：
- 新玩家第一次從開場到第一個事件。
- 只改第一條可玩路徑，不同時重設整個遊戲。

## 7. 製作原則

1. **先穩定，再加內容。**
2. **前台沉浸，後台可分析。**
3. **每個畫面只給一個主行動。**
4. **功能入口可以低調，但不能不可見或不可點。**
5. **歷史複雜性靠互動累積，不靠標籤講解。**
6. **手機/iPad 是獨立製作，不是桌面縮放。**
7. **任何 overlay 都必須有明確 layer policy。**
