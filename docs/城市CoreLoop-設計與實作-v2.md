## 1. 設計原則與核心循環

### 1.1 核心結論
證據**不**該是「點熱點自動收錄」，而是玩家完成一個短小歷史**查證**動作後的成果。

### 1.2 核心循環（現行兩段式）
```text
入城感受衝擊
→ 點熱點 → 完成 1–2 步輕量查證任務 → 收錄一張證據卡（+1 見識、解鎖該熱點的探訪事件）
→ 再點同一熱點「探訪 →」→ 事件（敘述 + 選項 payoff + 求知考驗 DSE）
```
**重點**：查證（任務）與探訪（事件）是兩段——第一次點做任務收證據，第二次點才探訪進事件（北京、上海一致）。

### 1.3 設計原則
- 低成本互動，**不要**倒數、速度、複雜失敗；答錯可重試，過關才收錄。
- 每張證據卡 = 一句可直接當 DSE 答案的句子。
- `findImage` 優先用**熱點自己的特寫圖**做「湊近查證」，比城市全景更沉浸（決議 2026-06-02）。
- 設施宜由**史事解鎖**（`facilityIsLocked`：`unlockYear` + 可選 `unlockEventId`），非純年份機械解鎖；未解鎖時「可見但鎖定」，文案「尚未設立／待史事後開啟」。
- 每城密度建議：3 核心任務 + 1–2 補充 + 2–4 設施 + 2–5 史事/行動。

## 2. 引擎與資料格式（程式現實）

### 2.1 流程與函式（intro.js）— v67 已打磨
```text
openHotspot()
  └ 有任務且未完成 → openEvidenceTask() → #evidenceTaskModal
        ├ 非 find 任務頂部顯示該熱點特寫圖 banner（#etBanner，與 find 視覺統一）
        └ 過關 → setTaskSubmitReady(true, success)（啟用 #etSubmit）
        └ 按「收錄證據」→ completeEvidenceTask()
              ├ markEvidenceTaskDone / collectEvidence(+1見識) / unlockedEvents.add
              └ showEvidenceReveal()：同一 modal 內呈現「證據卡」儀式
                  （朱砂「證」印 + 證據句 + DSE + ＋1見識）
                  └ 「探訪 →」直接 openEvent()（流程合一，免二次點熱點）
                     ｜ 若 optionalActionsLocked() → 顯示「要事待辦·暫不探訪」+「完成」鈕
```
打磨重點（v64–v67）：①流程合一（收錄→證據卡→探訪一氣呵成）②三型加情境 banner ③證據卡朱砂印儀式 ④classify：放入項淡化+「→分類」標籤、虛線投放區、選取發亮、兩步提示 ⑤sequence：洗牌+隱號。

### 2.2 四型任務（`renderEvidenceTaskBody` 依 `type` 派發）
| type | 玩家動作 | render 函式 | 過關條件 |
|---|---|---|---|
| `pick` | 數句選 1 | `renderPickTask` | 選到 `correct:true` |
| `classify` | items 分到正確 group | `renderClassifyTask` | 全部歸對 |
| `sequence` | 依正確先後點擊（**顯示已洗牌、隱藏序號**）| `renderSequenceTask` | 依原陣列順序逐一點對 |
| `findImage` | 在特寫圖點出正確區 | `renderFindImageTask` | 點中任一 `targets` |

> `sequence` v60 改良：洗牌+隱號＝真排序挑戰；`findImage` 為 v60 新增。

### 2.3 資料格式 `EVIDENCE_TASKS`（鍵 `'city:hotspotId'`）
```js
// 共通
{ type, title, instruction, prompt, evidenceText, dse, success }

// pick
options: [ { id, text, correct:true|false } ]
// classify
groups: [ { id, label } ],  items: [ { id, text, group:正確groupId } ]
// sequence（陣列順序即正確先後；顯示時自動洗牌）
steps:  [ { id, text } ]
// findImage
image:   'assets/hotspot/<id>.webp',     // 熱點自己的特寫圖
targets: [ { x, y, w, h } ],             // 正確區（百分比；點中即過）
decoys:  [ { x, y, w, h, hint } ]        // 誤區（點到給 hint，不過關）
```
**findImage 座標系**：`x/y/w/h` 為相對圖片的**百分比**（左上原點）；以 `img.getBoundingClientRect()` 換算點擊百分比比對。座標須對實圖在 preview 微調。

### 2.4 相關狀態（已納入 save/load）
`gameState.completedEvidenceTasks`(Set `city:hotspot`)、`foundHotspots`、`collectedEvidence`/`evidenceLedger`、`unlockedEvents`；證據門檻 `eventEvidenceCount`/`eventEvidenceRequired`。

## 4. 加一個新任務（步驟）
1. 確定熱點 id 與其 `unlocks` 事件（看 `CITY_SCENES[city].hotspots`）。
2. `EVIDENCE_TASKS` 加一條，鍵 `'city:hotspotId'`，照 §2.3 格式。
3. `findImage`：`image` 指向熱點特寫圖，先填粗略 `targets/decoys`，進 preview 用 `getBoundingClientRect()` 換算點擊驗證、微調座標。
4. 寫好 `evidenceText`（DSE 句）、`dse`、`success`。
5. 驗證：點熱點→任務→答對啟用收錄→收錄後 +1 見識→再點熱點可「探訪 →」。

> 內容（題幹／選項／證據句）原則上**先與設計者討論定稿**再寫入。

## 7. 成功標準
- 玩家入城知道要「查證」而非單純點熱點。
- 每張證據卡能直接轉化為 DSE 答案/短答句。
- 設施由史事推動解鎖，而非一開始任選。
- 史事因主圖更具吸引力（圖片階段達成後）。
