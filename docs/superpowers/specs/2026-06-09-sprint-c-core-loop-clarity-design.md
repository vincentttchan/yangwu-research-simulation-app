# Sprint C: Core Loop Clarity 設計規格

## 目標

Sprint C 的目標是讓玩家在第一輪遊玩中清楚理解遊戲的核心循環，而不是只被電影感或史事文字帶着走。玩家應能在 5–8 分鐘內完成一次完整循環：

`看見地圖目標 → 前往城市 → 觀察紅點 → 查證線索 → 收錄證據 → 進入史事抉擇 → 看見後果 → 回到地圖`

本 sprint 採用「核心循環優先，視覺只服務清晰度」方向。所有視覺改動都應回答一個問題：玩家是否更清楚知道自己正在做甚麼、為何要做、下一步去哪裏。

## 背景問題

### 1. 視覺風格尚未完全統一

登陸頁、人物選擇頁、四拍過場、事件頁各有氣氛，但目前仍有斷裂：

- 登陸頁偏暗金電影海報。
- 人物選擇頁偏淡色簡報，與整體沉浸感不一致。
- 四拍過場最接近目標風格，可作為參照。
- 事件頁仍偏紙本 worksheet，遊戲感不足。

Sprint C 不做大型美術重設，但會把第一輪關鍵畫面的語言拉近：暗金、朱砂、版畫、紙本、低透明遮罩、克制留白。

### 2. B1 教學減法過度

B1 把新手引導從長教學壓縮成三段幕僚提示，沉浸感提高，但核心操作因果未被建立。Sprint C 應把教學改成「分段式、情境內、逐步觸發」，而不是回到長篇說明。

### 3. Modal 優先級影響學習節奏

玩家前往北京或某些目標城市後，史事 modal 可能立即彈出，遮蓋城市場景和教學提示。這使玩家還未理解「城市紅點／查證／證據」流程，就被推入史事抉擇。

### 4. 紅點資訊重疊

紅點目前承載至少兩層資訊：場景所見、可互動入口。當兩者同時或過近呈現時，會互相遮擋，令學生不清楚應先讀、先點，還是已經進入任務。

### 5. 事件頁仍有考試感

事件頁需要承載歷史思考，但不應讓玩家感覺突然進入考卷。選擇應更像「案卷抉擇／處置路線」，結果應更像「局勢回應」，而不是「答題與評分」。

## 範圍

### 包含

- 第一輪核心路徑：李鴻章路線的開局、地圖第一步、上海／北京早段城市循環。
- 新手教學節奏與觸發時機。
- 城市紅點的 hover/tap、點擊、查證、收錄證據流程。
- 史事 modal 的開啟優先級與第一次入城延遲規則。
- 登陸頁、人物選擇頁、過場頁、第一個事件頁的局部視覺統一。
- 穩定性測試，防止顯性 worksheet / DSE / 長教學語氣回流。

### 不包含

- Supabase、登入、研究 logger、後台資料。
- 完整手機／iPad 重排。
- 新增大量歷史內容或新人物路線。
- 重做所有事件、所有城市或完整美術系統。
- 改變研究工具、評分 rubric 或 instruments 文件。

## 核心循環設計

### C1. 分段式幕僚提示

教學由三段增加至約六段，但每段只解釋一件事。提示語氣維持「幕僚低語／局勢提示」，避免「教學步驟 1、2、3」的介面說明感。

建議提示順序：

1. 朱砂印代表當前要務。
2. 青綠印代表可另行探訪。
3. 入城後先看紅點。
4. 紅點浮出的文字只是「所見」。
5. 點紅點才進入查證／證據。
6. 證據入卷後，史事抉擇才會展開。

提示應在玩家到達相應情境時才出現。地圖提示只講地圖；城市提示只講城市；紅點提示只講紅點。

### C2. 第一次入城保護

第一次進入某個新城市時，不應立即打開史事 modal，除非玩家已經完成該城市的第一個觀察／查證動作，或明確點擊史事入口。

建議規則：

- 若是第一次入城：先顯示城市場景、紅點和城市提示。
- 若該城有強制主線事件：先把目標標記在 mission sheet 或上方提示，不立即彈 modal。
- 玩家完成第一個紅點查證後，可自然接入主線事件。
- 若玩家嘗試離開或推進時間，才提醒「此城有要事未辦」。

這個規則的目的不是降低主線壓力，而是讓玩家先學會操作再承受歷史壓力。

### C3. 紅點資訊分層

紅點應分成三個清楚層次：

1. hover/tap：只浮出一句短「所見」，像現場觀察。
2. click：打開查證小面板，展示要查的線索或任務。
3. complete：短 toast 顯示「已收入手卷」，然後推動下一步。

紅點旁不應同時顯示兩個互相遮蓋的文字框。若 hover 提示存在，點擊後的查證面板應取代 hover 提示，而不是疊在上方。

### C4. 第一輪視覺統一

以四拍過場作參照，統一第一輪關鍵畫面：

- 登陸頁保留暗金、廢墟、大片黑罩，但主 CTA 可再清楚。
- 人物選擇頁降低淡色簡報感，增加暗色版畫氛圍和更強的角色視角。
- 四拍過場保留現有方向，只修排版與人物肖像融入度。
- 事件頁改為「案卷」而非 worksheet：選項像三條處置路線，結果像局勢回聲。

視覺統一不應犧牲可讀性。文字仍要在 iPad 與桌面上清楚可讀。

### C5. 事件頁遊戲感

事件頁的基本內容可保留，但呈現方式要改：

- 選項標籤從「抉擇」變成更具處置感的短句。
- 選項卡片不應像答題選項，應像奏摺、處置方案或案卷批示。
- 結果不強調正誤，而強調四方阻力、見識、人物關係或後續局勢的變化。
- 若需要歷史解釋，放在結果後的短「餘波」或「手卷補記」，避免打斷選擇節奏。

## 互動優先級

畫面同時存在多種提示時，優先級如下：

1. 流程型 modal：事件、證據任務、結算。
2. 教學 coachmark：只在沒有流程型 modal 時顯示。
3. 城市 mission sheet：作為下一步提示，不應遮住紅點。
4. 紅點 observation：短暫、就近、可被查證面板取代。
5. 地圖 topbar hint：只給方向，不承載長說明。

若流程型 modal 打開，coachmark 必須延後；若 coachmark 正在顯示，不應被自動彈出的史事 modal 打斷。

## 接受標準

- 玩家首次進入地圖後，能清楚知道第一個目標城市。
- 玩家首次入城後，先看到城市場景與紅點提示，而不是立即被史事 modal 遮住。
- 紅點 hover/tap 只顯示一層短觀察；點擊後不與查證面板重疊。
- 玩家能完成一次完整循環：城市觀察 → 查證 → 證據 → 史事抉擇 → 回到地圖。
- 新手提示不少於五段，但每段不超過兩句。
- 前台不出現 DSE 標籤、worksheet 語氣、顯性考試語言。
- 第一輪人物選擇頁、四拍過場、事件頁視覺語言更一致。
- `npm run check:syntax`、`npm run check:stability`、`npm run check:assets`、`npm run build` 通過。

## 驗證路徑

桌面優先檢查：

1. 登陸頁主 CTA 清楚，底部功能不搶焦點。
2. 李鴻章人物頁不再像簡報頁，主行動清楚。
3. 四拍過場完成後進入地圖。
4. 地圖目標城市清楚，且可探索城市不與主線目標混淆。
5. 進入第一個城市後，coachmark 不被事件 modal 立即遮蓋。
6. 紅點 hover/tap 不出現雙層遮擋。
7. 完成查證後自然進入或提示下一個史事。
8. 事件頁選項與結果更像案卷抉擇，而非考題。

iPad / mobile 待本 sprint 基本通過後再作第二輪檢查；若發現核心操作在觸控上不可用，應納入 Sprint C 修正。

## 後續實作原則

- 先修流程優先級，再修視覺。
- 先修第一輪核心路徑，再擴展到所有城市。
- 每個階段只增加必要提示，不把教學重新變成長說明。
- 任何視覺改動都要保留歷史沉浸感與可讀性。
- 不碰研究 logger/backend，避免與平行工作衝突。
