# Sprint C2: Core Loop Feel Polish 設計規格

## 目標

Sprint C2 的目標是在 Sprint C 已建立的核心流程上，改善第一輪遊戲的「感覺」。本 sprint 不再增加大量教學文字，而是讓玩家透過畫面位置、觸發節奏和視覺連續性，自然理解：

`紅點是現場線索 → 查證使線索成為證據 → 證據推開歷史事件 → 抉擇改變局勢`

玩家不應感到自己在完成一套教學表格，而是像在一個歷史場景中追查、判斷和處置。

## 背景問題

### 1. 紅點提示與觀察文字距離太遠

目前紅點旁有短標籤，而觀察文字可能出現在較遠位置。視線被拆成兩段後，玩家未必理解兩者屬於同一條線索。這削弱了「從現場痕跡深入查證」的連續感。

### 2. 完成北京入城教學後仍需再次進入北京

Sprint C 已避免史事太早遮蓋教學，但完成首次查證後，應立即把玩家帶入第一件歷史事件。若玩家仍要再點城市或再找入口，核心循環會中斷，像是完成了教學卻未真正開始遊戲。

### 3. 人物選擇頁左右割裂

人物選擇頁目前左側像資料板，右側像肖像展示。兩者雖然各自清楚，但未融合成同一個歷史場景，與登陸頁和四拍過場的電影感存在斷裂。

## 範圍

### 包含

- 第一輪城市線索：尤其是北京首次入城與第一個紅點。
- 紅點 hover/tap observation 的位置、尺寸、層級和與 evidence task 的過渡。
- 完成首次 evidence task 後，自動觸發該城市待辦史事的流程。
- 人物選擇頁的視覺整合：背景、肖像、文字、能力資料、CTA 的層級。
- 穩定檢查，防止核心流程回到「查證後無下一步」或「紅點提示遠離紅點」。

### 不包含

- 新增研究 logger 或後台資料。
- 重寫所有城市與所有事件。
- 完整 iPad / mobile 版面重排。
- 新增人物路線、劇情章節或大量新史料。
- 改動 Transfer Task、HEA、HNET 等研究文件。

## 設計方向

### C2.1 紅點就地展開

紅點互動應像「在現場靠近一個可疑痕跡」：

- hover/tap 時，只在紅點附近顯示短標籤和一句觀察。
- 觀察框應與紅點保持視覺連接，不應跨畫面遠距離呈現。
- 若紅點靠近畫面邊緣，觀察框可自動改到另一側，但仍要保持近距離。
- 點擊紅點後，觀察框收起，evidence task 由同一條線索延伸出來。

前台語氣仍保持情境化，不使用「步驟一／步驟二」或「題目」語感。

### C2.2 查證後立即接史事

首次進入北京這類主線城市時，流程應是：

1. 玩家進入城市。
2. 看見紅點與短觀察。
3. 點擊紅點完成查證。
4. 證據收入手卷。
5. 史事自動展開。

這個自動展開只應發生在「首次核心循環」或「已有明確待辦史事」的情況下。若玩家之後自由探訪城市，完成一般線索後可回到城市場景，不必每次都強制開史事。

### C2.3 人物選擇頁一體式電影構圖

人物選擇頁應改成「角色登場」而不是「角色資料表」：

- 背景與肖像需要融合成同一個歷史氛圍，避免左右像兩個獨立容器。
- 左側文字像片頭字幕：人物名、路線定位、短引句、簡短任務感。
- 右側肖像保留，但應以霧化、暗角、光影或紙紋融入背景。
- 能力與路線資料降級為輔助資訊，不應搶主視覺。
- CTA 保持清楚，但不要像表單提交按鈕；更像「擇此人物」或「啟此路線」。

文字總量不增加。若需要更多人物資料，應放入人物冊，而不是人物選擇首屏。

## 互動優先級

1. 玩家完成第一次 evidence task 後，若有被延遲的主線史事，優先打開史事。
2. 自動史事開啟前，應先給極短確認感，例如「已收入手卷」。
3. 紅點 observation、evidence task、史事 modal 不應同時重疊。
4. 人物選擇頁不應暴露過多遊戲數值，避免破壞片頭感。

## 接受標準

- 紅點觀察文字與紅點保持近距離，桌面畫面上不再分離到左右兩端。
- 點擊紅點後，觀察提示會收起，不與 evidence task 重疊。
- 玩家首次進入北京完成查證後，第一件史事自動展開，不需要再次點北京。
- 自動展開史事前，玩家能感到自己剛完成「證據入卷」。
- 人物選擇頁左右割裂感降低，背景、肖像、文字屬於同一個場景。
- 人物選擇頁保留清楚 CTA，玩家仍能迅速選定角色。
- 前台不新增顯性考試、worksheet、研究量表語氣。
- `npm run check:syntax`、`npm run check:stability`、`npm run check:assets`、`npm run build` 通過。

## 驗證路徑

桌面優先：

1. 登陸頁進入人物選擇頁。
2. 檢查李鴻章人物頁是否更像一體式角色登場，而非左右分裂資料板。
3. 選擇李鴻章並進入地圖。
4. 前往北京。
5. hover/tap 第一個紅點，確認短觀察貼近紅點。
6. 點擊紅點進入查證，確認 observation 不重疊。
7. 完成查證後，確認第一件史事自動展開。
8. 完成史事後回到地圖，確認核心循環仍然可繼續。

第二輪再檢查 iPad 與手機觸控距離；若桌面核心流程尚未穩定，不先擴大 responsive scope。

## 後續實作原則

- 先修互動連續性，再修大面積視覺。
- 先保障北京第一輪，再抽象到其他城市。
- 視覺只服務沉浸與理解，不新增裝飾性資訊。
- 不碰研究 logger、Supabase、登入或文件工作。
- 若現有 CSS 過多覆蓋導致人物頁難以微調，可只新增末端覆蓋，不做大規模清理。
